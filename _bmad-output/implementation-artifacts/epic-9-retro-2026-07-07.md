## Epic summary

Epic 9 replaced build-time `LANDING_*` env vars as the primary homepage source with a database-backed Site Page, operator-facing Website builder, and runtime public render at `/`. Operators can draft, preview, publish, brand the site, control homepage activities, and recover from a bad publish — without rebuilding the web container.

## What went well

- **Vertical slice sequencing** — Entity → API → cache → seed → public render → admin UI → branding → SEO → polish kept each story shippable and testable in isolation.
- **Singleton Site Page model** — Fixed singleton id + JSON draft/published payloads matched single-tenant AD-3 and avoided premature multi-site complexity.
- **Publish gate on server** — Hero headline, unpublished activity CTAs, and schema version checks prevented bad publishes even when client gate was stale.
- **Seed safety** — First-deploy seed + auto-publish preserved thesocialcollectivesg.com content; re-run does not overwrite operator draft.
- **Code review culture** — Adversarial reviews caught real operator pain (accent flush before save, `isDirty` on preset/revert, hero upload races, duplicate SEO fetches) before UAT.
- **Recovery features** — One-level revert snapshot and presets gave operators confidence without full version history.

## What we learned

- **Client/server draft sync is subtle** — Any server-side mutation (preset apply, revert) must respect `isDirty` the same way publish does, or local edits vanish silently.
- **“Preserves branding” must be explicit** — Logo at document root vs hero image in section props are different; UI copy and merge logic must list both.
- **Integration tests need CI stack** — Admin/public site tests exist but skip without Postgres/Redis; deferral accumulated across all 8 stories.
- **Preview security is acceptable MVP, not done** — Query-string preview tokens and JWT-shared HMAC key are documented defers; fine for single-tenant UAT, not long-term hardening.
- **Unsaved guard is project-wide debt** — Router navigation, sign-out, and session expiry still bypass builder guard (deferred from 9.5); Epic 9 did not solve holistically.

## Key insights

1. **Operator CMS in the CRM app works** — Draft/publish/preview in-dashboard matches how operators already manage activities; no separate CMS needed for v1.
2. **Redis cache + publish invalidation is the right freshness model** — Meets ≤30s target without rebuild; document for future editors.
3. **Code review on UI-heavy stories pays off** — Builder stories (9.5–9.8) had the highest patch counts; backend stories (9.1–9.3) were cleaner after first review.
4. **Revert one deep is enough for v1** — Full version history would be over-engineering; snapshot on publish matches AC and operator mental model.
