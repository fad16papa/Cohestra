# Epic 6 Retrospective — Post-MVP Enhancements

**Date:** 2026-06-22  
**Epic:** 6 — Post-MVP brownfield enhancements (catalogs, UX polish, campaign/outreach hardening)  
**Status:** Complete (stories 6.1–6.5)  
**Baseline commit:** `793a2cd`

## Epic summary

Epic 6 closed the gap between MVP ship and operator-ready daily use. Work was driven by real usage: campaigns blocked by missing consent, SendGrid delivery confusion, duplicate WhatsApp timeline noise, and the need for managed Communities/Categories instead of free-text labels.

### Stories shipped

| Story | Outcome |
|-------|---------|
| 6.1 | Communities & categories catalog — backend CRUD, migration seed, admin pages, nav submenu, community leads view |
| 6.2 | All Activities filters — category/community dropdowns backed by catalogs; server `community` query param |
| 6.3 | Form field editor — two-panel layout, fixed-height scroll, mobile stack / desktop split, bounded preview |
| 6.4 | Client profile UX — timeline scroll, WhatsApp follow-up dirty-state guard, toast deduplication, list alignment |
| 6.5 | Campaign hardening — consent on legacy templates, consent backfill migration, send-block reasons, SendGrid error parsing |

## What went well

- **Usage-driven scope** — Each story traced to an operator pain point (send disabled, duplicate follow-ups, unwieldy form editor, inconsistent labels).
- **Brownfield pattern held** — Catalogs seeded from existing activity labels avoided empty-state bootstrap on migration.
- **Minimal API extension** — Community filter on activities list matched existing category filter pattern without rewriting activity model.
- **UX guards without backend churn** — WhatsApp follow-up dirty baseline stopped timeline spam without a new API contract.
- **Fast iteration loop** — Form editor and filter dropdowns reused existing select styling and catalog fetch patterns from create-activity flow.

## What we learned

- **String labels vs catalog entities** — Shipping catalogs as admin-managed names (not FKs) was the right MVP+ tradeoff for speed, but rename/delete semantics must stay explicit in docs and deferred-work.
- **Consent is a data problem, not just UI** — Campaign send appeared broken until legacy registrations had consent fields and backfill; segment counts must match send path rules.
- **SendGrid “Processed” ≠ inbox** — Operators need clear messaging that API accept is not delivery; DNS/sender verification is an operator task outside the app.
- **Destructive actions need guard rails** — Save buttons that append audit timeline events should default disabled until the form is dirty (same pattern as publish/save elsewhere).
- **Post-MVP work deserves its own epic** — Tracking Epic 6 separately kept sprint-status honest after Epics 1–5 marked done.

## Challenges and growth areas

- **No formal PRD update** for Communities/Categories — behavior is implemented but planning artifacts still describe community as a string on Activity.
- **Server accepts duplicate follow-up POSTs** — UI guard only; malicious or API clients could still spam timeline.
- **Activity list still capped at 100 items** — Catalog filters help but pagination story remains open from Epic 2.
- **Zero automated regression** for Epic 6 paths — consistent with MVP defer pattern but risk rises as operator relies on platform daily.

## Key insights

1. **Production feedback belongs in tracked epics** — Brownfield fixes should get story IDs and retro closure, not only chat history.
2. **Audit append actions need “dirty or don’t save”** — Any button that writes timeline/history should mirror form dirty semantics.
3. **Catalog UX must be end-to-end** — Create flow + list filters + client filters must all pull from the same source or operators lose trust in labels.
