# Epic 1 Retrospective — Platform Foundation & Operator Access

**Date:** 2026-06-22  
**Epic:** 1 — Platform foundation, auth, design system, admin/public shells  
**Status:** Complete (stories 1.1–1.11)

## Epic summary

Epic 1 established the full-stack baseline: .NET 9 layered solution, Docker Compose, PostgreSQL + Redis, JWT auth, Next.js 16 + shadcn/ui, warm brand tokens, no-flash theming, and operator login with protected admin routes.

## What went well

- **Compose-first dev** — Single `docker compose up` path unblocked all later epics.
- **Design system early** — Tokens and theme before feature UI avoided hex drift and dark-mode bugs.
- **Clean architecture split** — Api / Application / Domain / Infrastructure / Contracts scaled through Epics 2–6 without re-scaffolding.
- **Auth contract frozen early** — `authFetch` + refresh rotation pattern reused everywhere.

## What we learned

- **Client-side route guard is MVP-acceptable** — Brief flash of admin shell before redirect; document until server middleware story exists.
- **Theme persistence in DB** — Normalizing empty `ThemePreference` to `system` at read time avoided migration churn.
- **OpenAPI + ProblemDetails** — Consistent error shape paid off in every subsequent API story.
- **Empty dashboard as gate** — Story 1.11 correctly deferred “real dashboard” until Epic 4; Epic 2 wired activity count later.

## Deferred carry-forward

From code reviews and `deferred-work.md`:

- No login/refresh rate limiting — NFR hardening story
- CORS `AllowedOrigins` hardcoded to localhost — deployment story
- `authFetch` session-expired toast shipped with Epic 2+ API usage
- Web healthcheck does not verify API connectivity — compose `depends_on` sufficient for MVP
- HTTP-only compose binding — local dev scope only
