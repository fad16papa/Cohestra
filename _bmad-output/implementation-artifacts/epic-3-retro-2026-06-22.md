## What went well

- **End-to-end capture loop** — QR → public form → Client in master list → profile/timeline.
- **Dedup without merge UI** — Merge-suspect flag matched MVP scope; avoided dangerous auto-merge.
- **Immutable registrations** — JSONB answers + append-only timeline aligned with NFR-8.
- **Public UX quality** — +63 phone default, sticky footer, live confirmation copy per UX-DR29.
- **Operator list ergonomics** — Server pagination/sort on clients list scaled better than activities list pattern.

## What we learned

- **Dedup is the epic risk center** — ATDD red phase for 3.3 was warranted; edge cases (email conflict, concurrent submit) stay in deferred-work.
- **Timeline projection is a product feature** — Merging registrations + audit events in `ClientTimelineBuilder` required careful ordering (Epic 5 extended types later).
- **Schema-bound answer history** — Answers outside current form schema omitted when schema exists; legacy data edge case documented.
- **List UX patterns echo forward** — Stale rows on refetch, no loading spinner on sort — same pattern appeared in Epic 4 reports.

## Deferred carry-forward

- No integration/unit tests for ingestion, dedup, timeline — Epic defer pattern
- Concurrent submits can duplicate clients — no unique DB constraint yet
- Unbounded registration lists on activity tab — pagination when volume grows
- Tab a11y (`tabpanel` wiring) — polish defer from Epic 2 pattern
- Referral extraction limited to `referral_source` field type
